1. Resumen ejecutivo

El presente Plan de Gestión del Proyecto describe el enfoque, los recursos, el cronograma, los roles, los riesgos y el proceso de seguimiento utilizados para la construcción del sistema EDI-301, una aplicación móvil destinada a apoyar la administración del programa institucional de Equipos de Desarrollo Integral (EDI) de la Universidad Linda Vista.

Este documento establece la planificación formal del proyecto, los mecanismos de control, los entregables principales, el equipo participante y los procesos de comunicación. Su objetivo es garantizar una administración eficiente del proyecto, asegurar el cumplimiento de los requisitos y proporcionar una base clara para la toma de decisiones durante el ciclo de vida del desarrollo.

2. Declaración del proyecto

El proyecto EDI-301 consiste en el desarrollo de una aplicación móvil multiplataforma (Android/iOS) utilizando Flutter, conectada a un backend REST en Node.js/Express y con base de datos SQL Server. La aplicación permitirá gestionar familias EDI, visualizar eventos, compartir publicaciones, administrar perfiles y fortalecer la comunicación con estudiantes y colaboradores por medio de un chat privado familiar.

El objetivo principal es digitalizar procesos actualmente manuales, reducir la carga administrativa del coordinador del programa y mejorar la experiencia de los estudiantes dentro de las familias EDI.

3. Alcance del proyecto

3.1 Objetivos del proyecto

  • Diseñar e implementar una aplicación móvil funcional para los usuarios del programa.
  • Desarrollar un backend seguro y escalable.
  • Integrar un sistema de autenticación seguro basado en JWT.
  • Ofrecer módulos claros: familias, eventos, publicaciones, perfil y notificaciones.
  • Proporcionar interfaces intuitivas y de fácil uso.
  • Entregar documentación técnica, de pruebas, arquitectura y manuales de usuario.

3.2 Productos del proyecto

  • Aplicación móvil EDI-301 (APK y build iOS).
  • API REST funcional.
  • Base de datos SQL Server estructurada y documentada.
  • Paquete de pruebas funcionales y reporte de resultados.
  • Manual de Usuario y Manual de Mantenimiento.
  • Documentos de Arquitectura, Diseño Detallado y Gestión de Configuración.
  • Plan de capacitación básico para el administrador del programa.

3.3 Exclusiones

  • Integraciones con sistemas institucionales no autorizados.
  • Desarrollo de una versión web completa del sistema.

4. Presupuesto del proyecto

Como proyecto académico y de implementación institucional, no existen costos de contratación externa. Sin embargo, se consideran recursos estimados en términos de horas de trabajo del equipo.

Análisis y levantamiento de requerimientos 25 h
Arquitectura y diseño técnico 20 h
Desarrollo frontend (Flutter) 60 h
Desarrollo backend (Node.js + SQL Server) 85 h
Pruebas y aseguramiento de calidad 50 h
Documentación 20 h
Despliegue y capacitación 10 h
Total aproximado de esfuerzo 270 h

6. Cronograma del proyecto

El cronograma se estructura en fases, adaptando las prácticas de planificación del documento de referencia al contexto específico del proyecto EDI-301.

Fase Actividades principales Duración estimada
1. Inicio y análisis Requerimientos, entrevistas, revisión de procesos, prototipos iniciales. 2 semanas
2. Diseño técnico Arquitectura del sistema, modelos de base de datos, diagramas UML. 1.5 semanas
3. Desarrollo backend Implementación de API REST, endpoints, seguridad y validaciones. 3 semanas
4. Desarrollo frontend Pantallas Flutter, navegación, manejo de estado, integración con API. 4 semanas
5. Pruebas Pruebas unitarias, funcionales, de regresión y aceptación de usuario. 2 semanas
6. Documentación Manual de usuario, manual de mantenimiento, reportes de pruebas, actualización de diseño. 1.5 semanas
7. Despliegue Configuración de entornos, carga inicial de datos, capacitación básica. 1 semana
Plazo total estimado ≈ 15 semanas

8. Plan de gestión de riesgos

Los riesgos identificados para el proyecto EDI-301 se resumen a continuación:

Riesgo Impacto Probabilidad Estrategia de mitigación
Retrasos por carga académica del equipo Medio Alta Reajuste del cronograma, priorizar tareas críticas, planificación anticipada.
Fallas en API o base de datos Alto Media Pruebas tempranas, monitoreo de logs, respaldos frecuentes.
Cambios tardíos en requerimientos Alto Media Validar alcance al inicio, documentar cambios y evaluar impacto.
Problemas con dispositivos móviles y versiones de SO Bajo Media Pruebas en varias versiones de Android/iOS y dispositivos de referencia.
Errores en almacenamiento o carga de imágenes Medio Media Validar rutas, tamaños y formatos; realizar pruebas específicas de subida y descarga.

9. Plan de seguimiento y control

El avance del proyecto se controlará mediante herramientas de gestión y reuniones periódicas.

  • Uso de GitHub Projects para la gestión de tareas y tableros Kanban.
  • Control de versiones mediante flujo Git (main, dev, feature, hotfix, release).
  • Revisión semanal del avance contra el cronograma definido.
  • Aplicación de pruebas de regresión antes de cada liberación importante.
  • Documentación continua de cambios relevantes en el sistema.

Los reportes de estado incluirán:

  • Avance por fase y tareas completadas.
  • Riesgos activos y su estado de mitigación.
  • Incidencias abiertas y cerradas.
  • Versiones liberadas y despliegues realizados.

10. Aprobación del proyecto

El presente documento constituye la base formal de gestión del proyecto EDI-301. Su aprobación implica que:

  • El alcance del proyecto está claramente definido.

  • Los roles y recursos han sido asignados.
  • Los riesgos principales han sido identificados y tienen estrategias de mitigación.
